## 已识别的主要瓶颈

### 1. `failures` 数据块仍然过大

虽然已经把原来的 `search` 重包拆开，但 `failures` 仍然接近 `200 kB`，这是当前最明显的静态数据体积压力点。

直接影响：

- 首次进入失败详情或结果页时会拉入较大的失败样本数据
- 数据继续增长后，结果页相关路径会先退化

### 2. 结果页仍按全量失败样本查询

当前 `ResultPage` 通过 `findFailuresByProgramId(program.id)` 过滤全量失败数据，逻辑简单，但扩展性一般。

风险：

- 数据量继续增长后，页面逻辑仍然建立在“先载全量，再过滤”的模型上
- 不利于后续做更细粒度的数据切片

### 3. 统计页与对比页仍直接依赖完整 `programs`

这是合理的当前实现，但如果后续 `programs` 规模继续增长，统计页和对比页也会逐渐成为重路径。

## 建议的后续优化顺序

### P1：对 `failures` 做切片发布

目标：

- 不再只生成单一 `src/data/failures.ts`
- 改为按 `programId` 或按学校维度输出多个轻量数据块

推荐方向：

1. 先按 `programId` 生成切片
2. 在结果页按目标动态加载该 program 对应失败样本
3. 失败详情页只加载当前失败记录所在切片

收益：

- 结果页从“加载全量失败样本”转为“加载当前目标相关样本”
- 对数据增长更稳定

代价：

- 需要引入异步数据读取
- 页面状态和加载态要补齐

### P2：把结果页改成按需异步取数

目标：

- 结果页不要默认依赖完整失败数据
- 从同步静态查找逐步转向“轻量程序索引 + 按需加载详情/失败数据”

建议拆法：

1. 保留 slug 定位逻辑
2. `program` 仍可从当前静态结构查到
3. `failures` 改为异步加载
4. 为结果页补 skeleton / loading / error state

### P3：评估 `programs` 是否需要二次拆分

当前 `programs` 约 `99 kB`，暂时还能接受，但如果数据持续扩容，后续可考虑：

- `programIndex` 保留首页搜索与浏览
- `programs` 进一步拆成“详情字段”和“统计字段”
- 对比页只拉所选 program 的详情数据

### P4：统计页预计算

当前统计页在运行时计算学校分布、专业分布、标签分布、分数区间。

后续可改为：

- 在 `sync-data.ts` 阶段直接预生成统计结果
- 前端只读预计算 JSON / TS 模块

收益：

- 减少运行时计算
- 让统计页逻辑更薄

### P5：引入真实的包体分析

当前只基于 Vite 输出做人工观察，没有正式的 bundle analysis。

后续建议：

- 加入构建分析脚本
- 固定观察首页、结果页、失败详情页三条路径的包体变化
- 把阈值纳入 CI 或人工 review checklist
